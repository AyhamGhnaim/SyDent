## ١. شو هو SyDent

نظام **SaaS متعدّد المستأجرين لإدارة عيادات الأسنان** موجّه للسوق السوري/العربي. كل طبيب (عيادة) بيسجّل وبياخد مساحة عمل معزولة خاصة فيه. وفي كمان **طبقة إدارة المنصّة** للمالك يدير منها كل العيادات (الاشتراكات، الخطط، الفوترة).

- **الواجهة:** عربي كامل، من اليمين لليسار (RTL)، خط Cairo، ثيم داكن + وضع فاتح، PWA (قابل للتثبيت كتطبيق).
- **طبقتان:**
  - **طبقة المستأجر (Tenant)** — ٢١ صفحة HTML للعيادات، كل عيادة معزولة عبر Row-Level Security بقاعدة البيانات (`owner_id = auth.uid()`).
  - **طبقة المنصّة (Platform)** — `admin.html`، إدارة SaaS بمستوى احترافي (مبنية على أنماط Stripe و Microsoft Partner Center).
- **صفحة عامة مستقلّة:** `book.html` — بوابة حجز إلكترونية للزوّار المجهولين. **تُعامَل كملف محمي مستقل — لا يُمسّ أبداً أثناء التعديلات الجماعية على باقي الصفحات.**

## ٢. التقنيات المستخدمة (Tech Stack)

| الطبقة | التقنية |
|---|---|
| الواجهة الأمامية | **HTML/CSS/JS صِرف — بدون أدوات بناء، بدون إطار عمل، بدون bundler.** كل صفحة ملف `.html` مستقل فيه JS مضمّن. |
| الوحدات المشتركة | `supabase-init.js`, `sidebar.js`, `theme.css` / `theme.js`, `timepicker.js` |
| الخلفية (Backend) | **Supabase** — PostgreSQL + Row-Level Security + Auth + Storage + دالة Edge واحدة |
| الاستضافة والنشر | **Cloudflare Pages**، نشر تلقائي من فرع `main` على GitHub عند كل push |
| إدارة الإصدارات | GitHub: `AyhamGhnaim/SyDent`، الفرع `main` |
| البريد الإلكتروني | **Resend** عبر custom SMTP (النطاق `mail.sydent.app`)، مربوط بـ Supabase Auth |
| المراقبة | BetterStack (uptime) + Sentry (أخطاء JS) |
| النسخ الاحتياطي | GitHub Actions ← نسخة قاعدة بيانات لريبو خاص + ملفات المرضى لـ Cloudflare R2 |

**مهم:** المنتج المنشور هو الكود **الصِرف (HTML/JS)** بجذر الريبو. ما في خطوة compile — عدّل ملف `.html`/`.js`، اعمل push لـ`main`، و Cloudflare Pages بينشره. (راجع القسم ١١ للملفات القديمة اللي **ليست** جزءاً من الإنتاج.)

## ٤. كيف يعمل النشر

1. اعمل push لِـ commit على `main` في GitHub.
2. **Cloudflare Pages** بيكتشف الـpush وبينشر الملفات الساكنة تلقائياً على `sydent.app`. بدون بناء، بدون أي إجراء يدوي.

### قاعدة كسر الكاش (Cache-busting)
بما إنو ما في bundler، الملفات المشتركة بتُرقَّم بـ query string `?v=YYYYMMDD[حرف]` على وسوم `<script>`/`<link>`. **التوكن الحالي: `20260624a`.**
لما تغيّر ملف مشترك (`supabase-init.js`, `sidebar.js`, `theme.css/js`, `timepicker.js`)، لازم ترفع التوكن في **كل ملفات HTML ما عدا `book.html`**، حتى يجلب المتصفّح النسخة الجديدة. هاد حالياً يُعمَل يدوياً — أتمتته من المهام المعروفة (القسم ٩).

### `keep-alive.yml`
الخطة المجانية لـ Supabase بتنيّم المشروع بعد ٧ أيام بلا أي استعلام، والمجانية بلا نسخ احتياطي تلقائي، فالمشروع النائم خطر. هالـworkflow بيعمل ping على REST API مرّتين بالأسبوع ليبقي المشروع صاحياً خلال فترة ما قبل الإطلاق. المفتاح اللي بيستعمله هو **المفتاح العام (anon)** (مو سرّ). خلّيه شغّالاً حتى الإطلاق.

## ٥. قاعدة البيانات والترحيلات (Migrations) — اقرأ هالقسم بتمعّن

**مصدر الحقيقة = مخطّط قاعدة البيانات الحيّ في Supabase، وليس ملفات الترحيلات.**

مجلد `migrations/` فيه ٦٠ ملف SQL، لكنه **سجلّ تاريخي جزئي**. خلال التطوير، كثير من تغييرات المخطّط طُبّقت **مباشرة في محرّر SQL بـ Supabase** ووُثّقت كملفات أحياناً فقط. أرقام الترحيلات فيها فجوات (مثلاً ١١، ١٢، ١٧–٢٣ بلا ملفات) وبعض الأرقام أُعيد استخدامها. **لا تفترض أن الملفات تعيد إنتاج المخطّط الحيّ.** لو احتجت تجهيز بيئة جديدة، اسحب المخطّط من قاعدة البيانات الحيّة بدل إعادة تشغيل هالملفات.

## ٦. الأسرار والمفاتيح — ⚠️ دوّرها كلها قبل أول عميل

**لا توجد قيم أسرار حقيقية في هذه الوثيقة ولا في الريبو، بشكل مقصود.** أدناه **مكان** كل سرّ ووظيفته. **بما أن المالك الأصلي غير مبرمج وأن المفاتيح شُورِكت مع أدوات الذكاء الاصطناعي خلال التطوير، يجب على المبرمج الجديد تدوير كل سرّ مذكور هنا قبل استقبال أي عميل حقيقي.**

### أسرار GitHub (الريبو ← Settings ← Secrets and variables ← Actions) — ٧ أسرار
| السرّ | الوظيفة |
|---|---|
| `SUPABASE_DB_URL` | سلسلة اتصال قاعدة البيانات للنسخ الاحتياطي اليومي |
| `BACKUP_PAT` | توكن fine-grained مقصور على ريبو النسخ الاحتياطي فقط |
| `SUPABASE_S3_ACCESS_KEY_ID` / `SUPABASE_S3_SECRET_ACCESS_KEY` | قراءة ملفات المرضى من تخزين Supabase (بروتوكول S3) |

بالإضافة لمتغيّر repo اسمه `BACKUP_ENABLED` (لازم يكون `true` ليشتغل النسخ الاحتياطي).

## ٧. النسخ الاحتياطي والاستعادة

- **قاعدة البيانات:** GitHub Actions `backup.yml` يشتغل يومياً (الساعة ٢ صباحاً)، يأخذ نسخة من القاعدة (الأدوار + المخطّط + البيانات، بما فيها مخطّط `auth` فحسابات الدخول محفوظة) إلى الريبو الخاص `AyhamGhnaim/SyDent-backups` بمجلد مؤرّخ.
- **ملفات المرضى:** تُزامَن يومياً من تخزين Supabase (`patient-files`) إلى حاوية Cloudflare R2 اسمها `sydent-backups` عبر rclone (نفس الـworkflow).
- **الاستعادة:** موثّقة بشكل منفصل في `SyDent_DR_Runbook.md` (ضمن ملفات سياق المالك). **لم يُجرَ اختبار استعادة كامل حتى الآن — اعمله مبكراً كمالك جديد** للتأكد أن النسخ قابلة للاستعادة فعلاً.

## ١٠. العمل المتبقّي — مصنّفاً

### 🔴 إلزامي قبل أول عميل حقيقي
1. **دوّر كل الأسرار** (القسم ٦). شُورِكت مع أدوات ذكاء اصطناعي خلال التطوير. يشمل GitHub PAT، مفتاح Resend API، مفتاح Supabase service-role، ومفاتيح S3/R2 للنسخ الاحتياطي.
2. **درّج DMARC.** البريد شغّال عبر Resend؛ DMARC حالياً على `p=none` (مراقبة فقط) في سجل DNS `_dmarc.mail.sydent.app`. بعد التأكد أن تقارير DMARC تُظهر مرور كل البريد الشرعي، شدّد إلى `p=quarantine` ثم `p=reject` (سجل TXT في Cloudflare DNS). هذا يمنع انتحال بريدك أو وقوعه في السبام.
3. **قرار سعة البريد.** حدّ معدّل بريد Supabase Auth مضبوط على **٤/ساعة** ليطابق الخطة **المجانية** لـ Resend (١٠٠/يوم). كافٍ لعدد قليل من العيادات لكنه سيخنق التسجيل المتزامن عند التوسّع. قبل استقبال عملاء كثر، رقِّ Resend (Pro ≈ ٢٠$/شهر = ٥٠ ألف إيميل) وارفع حدّ معدّل Supabase.
4. **اعمل اختبار استعادة نسخة احتياطية** (القسم ٧) — لم يُجرَ أبداً.

### 🟡 تحسينات / قرارات
- **منطقة Resend:** حالياً US East (منطقة EU كانت تحتاج خطة مدفوعة وقت الإعداد). أعد النظر لو يهمّ زمن الوصول/الامتثال.
- **نفق Sentry:** حاجبات الإعلانات (مثل Brave) تحجب CDN تبع Sentry؛ تمرير الأحداث عبر نطاق SyDent يعطي تغطية أخطاء ١٠٠٪. اختياري.
- **`admin.html` يستخدم `alert()` ~١٥١ مرة** — يعمل، لكن تحويله لإشعارات toast تجميل اختياري.

### 🧹 نظافة الكود / التنظيف
- **PAT النسخ الاحتياطي `sydent-backup` بلا تاريخ انتهاء** — استبدله بتوكن least-privilege ينتهي بتاريخ.
- **H2 (XSS نظري):** أسماء المرضى تُدمج في template literals بـ`appointments.html` (`renderList`/`renderDay`). مسارات العرض الأخرى تهرّب عبر `escapeHtml`؛ هاتان النقطتان تحتاجان تدقيقاً وتصليباً. خطر نظري؛ لا استغلال معروف.
- **حلّ خطأ الاختبار `SYDENT-WEB-1`** في Sentry (تجميلي).
- **ملفات قديمة للمراجعة/الإزالة** (راجع القسم ١١).

### 🤖 الأتمتة (المحور ٥)
- **سكربت واحد لرفع توكن كسر-الكاش** عبر كل ملفات HTML (حالياً يدوي عبر ~٢١ ملف).
- **حُرّاس CI** للثوابت التطويرية (فحص صياغة JS، توازن div، ترتيب CDN، فحص الأسرار) التي كانت تُشغَّل يدوياً خلال التطوير.

### 🧪 اختبارات حيّة متبقّية (مخاطرة منخفضة، تأكيد سلوك فقط)
- زرع الترحيل ٧٠ التلقائي: حساب فارغ جديد ← صفحة العلاجات ← القائمة الافتراضية تُملأ تلقائياً بسعر ٠.
- شريط انقطاع النت: DevTools ← Network ← Offline ← يظهر شريط أحمر؛ Online ← يختفي.
- اختبار "Bug #1" (مدير تحويل الدخل): يحتاج سيناريو عدم حضور حقيقي + دفعة غير مكتسبة.

## ١١. نظافة الريبو — ملفات قديمة / غير إنتاجية (راجِع، لا تفترض)

موجودة بالريبو لكنها **ليست جزءاً من المنتج الشغّال**. راجِعها وغالباً احذفها:

## ١٣. مفاهيم أساسية في النطاق (حتى لا تكسر المالية)

- **المحاسبة على الأساس النقدي.** "split" الدفعة يُحتسب إيراداً مكتسباً فقط إذا كان `is_unearned = false` **و** مرتبطاً بجلسة **مكتملة**. توجد ثوابت حفظ (conservation invariants) واختبار ذاتي مدمج (Ctrl+Shift+D على أسطح المحاسبة).
- **منطق المالية/المحاسبة هو أكثر جزء حسّاس في النظام.** دُقّق بعمق. لا تعدّله بدون تحليل عميق — منطق توزيع الدفعات (FIFO) مكرّر عبر ثلاثة ملفات (`patient-profile.html`، `appointments.html`، `settings.html`) ولازم يبقوا متطابقين. مسار الحفظ الذرّي يمرّ عبر الـRPC `realloc_patient_splits`.
- **نموذج الأمان:** RLS لكل مستأجر (`owner_id = auth.uid()`). `book.html` هو السطح المجهول الوحيد ويكتب فقط عبر ثلاث دوال `SECURITY DEFINER` بحدود إساءة؛ جدوله غير ذلك deny-all للمجهولين.
